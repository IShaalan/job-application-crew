# Bullet Writing Guide

Strong resume bullets follow a consistent pattern: **strong verb + specific deliverable + quantified impact**. This guide shows what works, what fails, and why.

## General Rules

- **25 words max** per bullet. If it is longer, it contains two ideas — split or cut.
- **Every bullet ends with a period.**
- Lead with what you did, not what the team did.
- Metrics beat adjectives. "Increased retention 18%" beats "significantly improved retention."
- Avoid filler words: passionate, excited, leverage, utilize, various, helped.

## Product Manager Bullets

```
"Launched self-serve analytics dashboard from 0 to 1, adopted by 200+ enterprise customers within 6 months."
  -> Strong verb (launched) + specific deliverable (dashboard) + adoption metric + time-bound scope
```

```
"Reduced data pipeline latency by 65% by partnering with infrastructure team to redesign ingestion architecture."
  -> Quantified improvement + cross-functional signal + technical credibility
```

```
"Defined API-first integration strategy connecting 15+ third-party tools, reducing customer onboarding time by 40%."
  -> Strategic ownership + scope signal (15+ tools) + customer-facing outcome
```

```
"Built experimentation framework running 30+ A/B tests per quarter, increasing trial-to-paid conversion by 22%."
  -> System-building (not just running tests) + throughput metric + business outcome
```

```
"Conducted 80+ customer interviews and synthesized insights into a prioritization framework adopted across 3 product lines."
  -> Discovery rigor + artifact created + organizational adoption scope
```

```
"Owned product roadmap and quarterly OKR process, aligning 3 engineering squads to company-level revenue targets."
  -> Roadmap ownership + process leadership + cross-team alignment signal
```

## Software Engineer Bullets

```
"Built RESTful APIs serving 2M+ requests per day supporting the core SaaS platform."
  -> Clear deliverable + scale metric + business context
```

```
"Reduced deployment time from 45 minutes to under 5 minutes by implementing CI/CD pipeline with automated testing."
  -> Before/after metric + specific technical solution
```

```
"Designed event-driven notification system reducing alert latency from minutes to under 5 seconds."
  -> Architecture decision + quantified improvement with before/after
```

```
"Migrated 3 legacy services to Kubernetes, eliminating 12 hours of monthly manual infrastructure maintenance."
  -> Scope (3 services) + technology + time-saved metric
```

```
"Led incident response for P0 outage affecting 50K users, restoring service in 23 minutes with permanent fix shipped same day."
  -> Severity context + response speed + resolution completeness
```

```
"Introduced code review standards adopted by 40-person engineering org, reducing production bugs by 30% quarter over quarter."
  -> Process improvement + adoption scope + measurable outcome
```

## Designer Bullets

```
"Redesigned onboarding flow reducing time-to-first-value from 8 minutes to under 2 minutes for new users."
  -> Specific flow + before/after metric + user impact
```

```
"Created design system with 120+ reusable components, adopted across 4 product teams and cutting UI development time by 35%."
  -> Deliverable + scope (120+ components, 4 teams) + efficiency outcome
```

```
"Led user research program conducting 60+ interviews, uncovering 3 key pain points that reshaped the Q3 product roadmap."
  -> Research rigor + specific findings count + business impact (roadmap change)
```

```
"Increased mobile checkout completion rate by 28% through iterative prototyping and usability testing over 6 sprints."
  -> Business metric + methodology shown + time investment
```

```
"Designed accessibility-first dashboard meeting WCAG 2.1 AA standards, expanding addressable market by 15%."
  -> Specific standard + business justification for accessibility work
```

```
"Ran 12 concept tests validating pricing page redesign before engineering investment, saving estimated 3 engineering-weeks."
  -> Validation rigor + resource efficiency + de-risking signal
```

## Anti-Patterns

### 1. Vague impact, no metrics

```
BAD:  "Helped improve the user experience across multiple products."
  -> No metric, "helped" is passive, "multiple" is vague

GOOD: "Redesigned checkout flow across 3 products, improving task completion rate by 24%."
  -> Specific scope + measurable outcome
```

### 2. Too long (two ideas crammed into one bullet)

```
BAD:  "Led a cross-functional team of engineers, designers, and data scientists to build a real-time analytics
       dashboard that was adopted by over 200 enterprise customers and reduced support tickets by 35% within
       the first quarter of launch."
  -> 43 words. Contains launch story + support impact — split into two bullets.

GOOD: "Led cross-functional team to launch real-time analytics dashboard adopted by 200+ enterprise customers."
GOOD: "Reduced support tickets 35% in first quarter post-launch through self-serve analytics capabilities."
```

### 3. Responsibilities instead of results

```
BAD:  "Responsible for managing the product backlog and coordinating with engineering teams."
  -> Describes the job description, not what was accomplished

GOOD: "Prioritized backlog across 3 squads using RICE framework, shipping 40% more features per quarter."
  -> Same work, framed as outcome
```

### 4. Buzzword overload

```
BAD:  "Leveraged cutting-edge AI/ML technologies to drive innovative solutions that transformed the customer
       experience paradigm."
  -> Zero specifics, pure filler

GOOD: "Deployed ML-based recommendation engine increasing average order value by 18% across 500K monthly users."
  -> Specific technology + specific metric + scale
```
